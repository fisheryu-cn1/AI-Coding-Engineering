# AI 代码生成上下文控制：设计分析框架

> **目的**：为「小模型、大项目」场景下，AI 辅助代码生成的上下文控制问题提供一个结构化分析框架，用于指导后续方案设计与验证。  
> **核心矛盾**：小模型可用上下文窗口远小于理论最大值，但大型项目又要求全局一致性。  
> **两个目标**：(1) 稳定控制每一步任务复杂度并保持整体一致性；(2) 合理控制上下文焦点，避免冗余与遗漏。

---

## 一、问题背景与核心矛盾

### 1.1 观察到的现象

| 现象 | 根因推断 | 后果 |
|------|---------|------|
| 小模型在任务规模过大时错误率陡增 | 注意力稀释、推理深度不足 | 生成代码逻辑混乱、接口不匹配 |
| 上下文接近最大窗口时质量下降 | 有效可用窗口 < 理论窗口 | 即使未报错，输出质量已受损 |
| 缺乏全局信息时局部代码偏离整体设计 | 模型仅见树木不见森林 | 架构腐化、技术债务隐性积累 |
| 上下文塞满无关信息时关键约束被忽略 | 信号被噪音淹没 | 生成代码违反核心约定 |

### 1.2 核心矛盾矩阵

```
              上下文需要足够大
                    ↑
                    │
    全局一致性要求 ──┼── 局部精确性要求
    (需要宏观信息)   │   (需要聚焦细节)
                    │
                    ↓
              上下文需要足够小
              (模型有效处理能力有限)
```

**本质**：这不是简单的"大小"问题，而是**信息组织与注意力分配**的问题。

### 1.3 外部研究与实践观察

> 本节记录当前学术界和工业界对类似问题的研究发现，为框架中的假设提供外部佐证。

**（1）有效上下文窗口远低于标称值**

Paulsen (2025) 提出 **Maximum Effective Context Window (MECW)** 概念，通过实证研究发现：
- 所有模型的有效上下文窗口都**显著低于**标称的最大值，在复杂任务中差距可达 **99%**。
- GPT-4.1 (MCW=128k) 在检索任务中 MECW ≈ 15k tokens，但在复杂多步排序任务中仅 ≈ **1k tokens**。
- **任务类型决定真实性能**，而非 token 数量。简单查找任务可容忍较长上下文，多步推理任务的有效窗口急剧收缩。

> **对框架的启示**：不能依据模型的标称上下文上限设计方案，必须针对**具体任务类型**测量有效窗口，并预留安全余量。

**（2）"迷失在中间"与位置偏见**

Liu et al. (2024) 的 "Lost in the Middle" 研究表明：
- LLM 对位于上下文**中间位置**的信息回忆能力显著弱于开头和结尾。
- 不同模型表现出不同的位置偏见模式（Hutter et al., 2025）。
- 缓解策略包括：关键信息重排序（放置于首尾）、提示压缩、外部摘要模块、注意力重新加权。

> **对框架的启示**：全局约束（如架构契约）如果 buried 在上下文中部，可能被模型忽略。需要设计"锚定机制"确保关键约束始终处于显著位置。

**（3）注意力稀释（Attention Dilution）**

Transformer 的注意力机制随上下文增长而稀释：
- Softmax 归一化导致每 token 的注意力权重随上下文增长而缩小。
- 10万 token 上下文对应 100 亿对注意力计算关系，信号未增强，噪音基底上升。
- SkillReducer (2026) 的大规模实证（55,315 个公开技能）发现：**60%+ 的技能体内容不可操作**，移除这些噪音后功能质量反而提升 2.8%，验证了"少即是多"效应。

> **对框架的启示**：上下文组装不是信息越多越好，而是**信号噪音比**至关重要。需要主动剔除低价值内容。

**（4）仓库级代码生成的上下文困境**

InlineCoder (2026) 指出：直接将整个仓库喂给 LLM 不可行，因为 irrelevant/redundant code 会淹没关键信息。
- 传统 RAG 基于表面相似度检索，但代码中"相似 ≠ 相关"。
- **InlineCoder 方案**：生成草稿 → 双向内联（上游调用方+下游被调用方）→ 将仓库级生成转化为函数级任务，在 RepoExec 基准上相对提升 **29-49%**。

CodeCompass (2026) 提出 **Navigation Paradox（导航悖论）**：
- 更大的上下文窗口不会消除结构导航需求，只是将失败模式从"检索容量不足"转移到"导航显著性不足"。
- 架构关键但语义疏远的文件，即使上下文容量充足，也可能因注意力分配问题被忽略。
- 图导航（IMPORTS/INHERITS/INSTANTIATES 边）在架构密集型任务中显著优于纯检索。

> **对框架的启示**：单纯扩大上下文窗口或改进向量检索都不够，必须引入**结构导航机制**（如调用图、依赖图）来引导注意力分配。

**（5）多 Agent 协作与任务分解的工业实践**

Deep Agent Architecture (2026) 采用强制分层：
- **Orchestrator**：无代码访问权，只做战略规划与任务分解。
- **Explorer**：只读权限，负责调查与验证。
- **Coder**：实现权限，接收精炼后的上下文。
- 关键洞察：通过**强制隔离**防止"边探索边实现"的反模式，每个 Agent 仅接收其角色所需的蒸馏知识。

ProjectGen (2025) 提出迭代式项目级代码生成：
- ArchAgent 生成架构 → JudgeA 评估 → SkeletonAgent 生成骨架 → JudgeS 评估 → 迭代优化。
- 长上下文管理策略：仅保留与当前反馈最语义相似的 **γ** 条历史摘要，平衡历史意识与计算效率。

Cursor 用户实践观察 (2025-2026)：
- 大型复杂企业代码库中，自动上下文检索仍显"浅层"，Agent 仅做少量引用查找。
- 用户 workaround：维护 `memory.md` 文件描述项目结构，供 AI 每次读取。
- 这间接证实了**显式结构描述**比隐式检索更可靠。

> **对框架的启示**：任务分解不仅是"把大任务拆小"，更是**上下文隔离与蒸馏**的过程。不同角色的 AI 实例应看到不同的上下文视图。

**（6）AST 与静态分析：外部工具预处理 vs LLM 原生推理**

"Do Code LLMs Do Static Analysis?" (Su & McMillan, 2026) 的实证研究揭示了一个关键结论：
- **LLM 不擅长做静态分析**。在调用图和数据流图生成任务上，GPT/Gemini 的 Jaccard 相似度仅 **0.12–0.19**，链式准确率极低。
- **AST 生成相对容易**（标签学习），但 callgraph/dataflow 需要复杂推理，LLM 难以胜任。
- **上下文学习（in-context learning）只能改善 AST 生成**，对调用图/数据流生成几乎无帮助。

与此同时，AST 驱动的上下文工程取得了显著进展：
- **HASTE (2025-2026)**：Hybrid AST-guided Selection with Token-bounded Extraction，将 AST 结构感知与相关性检索融合。在 85% 代码压缩率下仍保持高结构保真度，解决了纯结构方法（保留完整函数但语义无关）与纯相关方法（找到相关行但切断依赖）之间的 trade-off。
- **cAST (EMNLP 2025)**：基于 AST 的结构感知分块（chunking via AST），替代传统的行级分块启发式，Recall@5 提升 4.3 点，SWE-bench Pass@1 提升 2.67 点。
- **AstraAI (2026)**：将 AST 分析与 RAG 结合，通过 clang-query 提取类成员变量、访问修饰符、函数签名和源码范围，比纯文本检索更精确。
- **STALL+ / SynthCoder**：静态分析在三个阶段注入——prompt 阶段提取文件级 API 签名和有效符号 token；decoding 阶段约束生成合法 API 调用（将非法 token 概率置零）；post-processing 阶段用编译器验证输出。

> **对框架的启示**：
> 1. **AST 解析、调用图构建、数据流分析应由外部专用工具完成，LLM 消费其结果**。让 LLM 自己推理代码结构是低效且不可靠的。
> 2. **AST 是连接原始代码与压缩上下文的理想中间层**。通过 AST 可以：精确提取签名而舍弃方法体、识别语法边界避免"Frankenstein context"、基于控制流/数据流进行程序切片、将代码结构与自然语言设计文档对齐。
> 3. **AST 剪枝和压缩是平衡焦点与背景信息的核心技术**，后续方案应深度整合 AST 驱动的上下文组装。

## 二、核心问题域定义

将问题拆解为两个正交维度：

### 2.1 纵向维度：任务分解与一致性（Task Decomposition & Coherence）

**问题**：如何将一个大型开发任务分解为一系列小模型可稳定处理的子任务，同时保证所有子任务的输出在整体上自洽？

关键子问题：
- 分解的粒度与边界如何确定？
- 子任务之间是否存在隐式依赖？如何显式化？
- 前序任务的输出如何约束后序任务而不造成上下文膨胀？
- 谁来承担"架构守卫"角色，防止局部优化破坏全局设计？

### 2.2 横向维度：上下文组装与焦点控制（Context Assembly & Focus）

**问题**：针对每一个具体的子任务，如何从项目知识库中筛选、组装、呈现最相关的上下文信息？

关键子问题：
- 哪些信息属于"必须始终携带"？哪些属于"按需加载"？
- 如何量化"信息相关性"？
- 如何在上下文长度受限时进行信息取舍？
- 当任务推进时，上下文焦点如何平滑迁移？

## 三、分析框架：四个关键维度

### 维度一：任务复杂度分层模型（Vertical Layering）

将 AI 代码生成任务按抽象层次划分为不同层级，每一层对应不同的上下文策略：

| 层级 | 名称 | 典型内容 | 上下文特征 | AI 角色 |
|------|------|---------|-----------|---------|
| **L0** | 战略层 | 项目愿景、核心目标、非功能性需求（安全、性能）、技术选型约束 | 极小但绝对不变，应常驻 | 架构守护者 |
| **L1** | 架构层 | 模块划分、接口契约、数据流、依赖关系、设计模式 | 相对稳定，按模块按需加载 | 结构设计师 |
| **L2** | 战术层 | 类/函数设计、算法逻辑、异常处理、测试策略 | 随任务变化，高度相关 | 实现工程师 |
| **L3** | 执行层 | 具体代码行、变量命名、注释、单测用例 | 高度聚焦，几乎纯局部 | 代码工匠 |

**分析要点**：
- 高层（L0-L1）信息必须**以某种形式**渗透到每一次低层调用中，但不能以原始体积渗透。
- 需要研究：高层信息的"压缩表示"（如架构契约摘要、设计约束清单）如何生成与维护。

**可参考方案**：
- **ProjectGen (2025)**：通过 ArchAgent 生成 SSAT（语义软件架构树），作为 L1 层契约，经 JudgeA 多维度评估（需求覆盖、接口一致性、依赖无环）后锁定。
- **Deep Agent Architecture (2026)**：Orchestrator 仅管理 L0-L1 层战略，Coder 仅接收 L2-L3 层实现上下文，强制隔离防止上下文污染。
- **用户实践 `memory.md`**：由 Agent 维护的项目结构文件，本质上是 L1 层的轻量手动摘要。

**建议分析方法**：
- 对现有项目人工编写 L0-L2 层"理想摘要"，测量其 token 数，评估目标模型的有效窗口是否能容纳「摘要 + L3 任务描述 + 直接相关代码」。
- 对比"有摘要约束" vs "无摘要约束"的生成结果一致性。

### 维度二：上下文信息分类（Information Taxonomy）

对进入 AI 上下文的所有信息进行分类，指导取舍策略：

| 类别 | 定义 | 示例 | 加载策略 |
|------|------|------|---------|
| **全局不变量 (Global Invariants)** | 任何任务都不应违反的底层约束 | 命名规范、错误处理策略、安全红线、核心数据结构定义 | **始终携带，高度压缩** |
| **直接相关量 (Direct Context)** | 与当前任务直接相关的定义和实现 | 正在修改的函数、其调用方/被调用方、依赖的类 | **完整加载，优先级最高** |
| **间接相关量 (Indirect Context)** | 通过关系链与当前任务关联的信息 | 间接依赖的服务、共享的工具类、配置文件 | **按需摘要，关系越远粒度越粗** |
| **历史依赖量 (Historical Dependency)** | 前序任务已生成的输出 | 已实现的函数、已定义的接口 | **按需引用，避免重复加载全文** |
| **环境约束量 (Environmental Constraints)** | 技术栈、框架版本、构建规则 | 语言版本、框架 API、ORM 映射规则 | **引用标识，详细定义外置** |

**分析要点**：
- 需要建立一套"相关性衰减函数"：随着与当前任务的距离增加，信息的详细程度应递减。
- 需要研究：如何自动判断一条信息属于哪一类。

**可参考方案**：
- **SkillReducer (2026)**：将技能内容分类为"可操作核心规则"与"按需加载补充内容"，通过 taxonomy-driven classification 和 progressive disclosure 实现 39% 体压缩，同时提升功能质量。
- **InlineCoder (2026)**：直接相关量 = 目标函数的签名+局部导入；间接相关量 = 通过 AST+LLM 预测的下游被调用方；通过草稿（anchor）动态确定需要哪些下游上下文。
- **Knowledge Graph Based Code Generation (2025)**：通过图遍历获取 n-hop 子图作为上下文，再经语义重排序和 top-k 剪枝控制间接相关量粒度。
- **AST 驱动的信息分级压缩**：
  - **直接相关量（完整代码）**：当前正在编辑的函数/方法体，通过 AST 确保语法完整性。
  - **直接相关量（签名级）**：调用方与被调用方的完整签名（参数、返回类型、异常），通过 AST 提取而无需加载方法体。参考 TestBench / APT 的代码简化策略：移除无关方法体但保留签名。
  - **间接相关量（摘要级）**：通过 AST-based logic summarization 提取控制流（if/for/while/try 路径）、数据流（def-use 链）和调用频率，形成结构化自然语言摘要。
  - **环境约束量（引用级）**：仅保留 import/using 的符号名称，通过 AST 提取顶层依赖声明。
- **程序切片 (Program Slicing)**：AutoBug (2025) 应用 Weiser 反向切片，移除不影响后置条件的语句，在保持验证条件的前提下显著减少 prompt 大小。

**建议分析方法**：
- 采用 SkillReducer 的两阶段思路：先分类（routing layer），再按需展开（progressive disclosure）。
- 为每个信息类别定义明确的 token 预算上限，形成"预算分配表"。
- **AST 压缩实验**：对同一任务，对比三种上下文形式（a）原始代码全文（b）AST 提取的签名+当前函数体（c）HASTE 式混合压缩，测量生成正确率与 token 消耗。

### 维度三：上下文焦点模型（Focus Model）

定义每一次 AI 调用的"上下文焦点"，类比摄影的景深：

```
[焦点中心] ←── 最清晰、最完整 ──→ [边缘] ←── 最模糊、仅摘要 ──→ [外部] ←── 不出现
   当前任务          直接依赖              间接依赖                  无关信息
  (100% 细节)      (80% 细节)           (20% 细节)                  (0%)
```

**关键参数**：
- **焦点半径 (Focus Radius)**：以当前任务为中心，多大范围内的直接相关量需要完整加载？
- **关系深度 (Relationship Depth)**：在依赖图/知识图中，向几跳之外探索间接相关量？
- **信息粒度 (Granularity)**：对于不同距离的信息，使用原始代码、接口签名、自然语言摘要，还是仅保留名称？

**分析要点**：
- 这三个参数应根据任务类型动态调整（如重构任务需要更大的焦点半径，新增小功能可以更聚焦）。
- 需要研究：是否存在一个可计算的最优焦点配置，或必须通过启发式/反馈调整。

**可参考方案**：
- **InlineCoder**：焦点半径 = 当前文件 + 直接引用的跨文件依赖；关系深度 = 1跳（上游调用方内联+下游被调用方检索）；信息粒度 = 上游用内联代码（完整），下游用检索到的函数体（完整）。
- **CodeCompass (2026)**：通过 MCP 暴露代码图（IMPORTS/INHERITS/INSTANTIATES），让 Agent 主动导航而非被动接收。图导航在 G2（结构可达）和 G3（隐藏依赖）任务上显著优于 BM25 和纯向量检索。
- **Graph-Structured RAG**：用 BFS/DFS 或个性化 PageRank 从当前节点向外扩展，根据任务类型调整深度（API 理解可能需要 2-3 跳，bug 修复可能需要全图可达性分析）。
- **AST 引导的焦点控制**：
  - **cAST 结构感知分块**：用 AST 边界（函数、类、代码块）替代行级分块，确保每个 chunk 是语法自洽的最小单元，避免检索到"半个函数"。
  - **HASTE 的 token-bounded extraction**：先通过 hybrid 检索定位相关 AST 节点，再沿 AST 父子关系扩展确保结构完整，最后按 token 预算进行语法感知压缩（优先保留接口签名，压缩方法体内实现细节）。
  - **SimAST-GCN**：简化 AST（删除不含连接属性的冗余节点）后构建关系图，降低上下文体积同时保留核心结构信息。

**建议分析方法**：
- 为不同任务类型（新增功能、重构、bug 修复、测试生成）分别实验最优焦点参数组合。
- 建立"焦点质量反馈环"：分析生成代码实际引用了哪些上下文，反推焦点配置是否合理。
- **AST 焦点对比实验**：对同一 bug 修复任务，对比 (a) 文件级上下文 (b) AST 提取的调用链上下文 (c) 向量检索的相似代码上下文，测量修复成功率与编译通过率。

### 维度四：一致性保障机制（Consistency Assurance）

在多步骤、多调用的生成过程中，如何防止"漂移"？

| 阶段 | 机制 | 承担者 | 关键问题 |
|------|------|--------|---------|
| **事前** | 约束注入 (Constraint Injection) | Prompt 工程 / 系统提示 | 如何将 L0-L1 的约束以不可忽略的方式写入每一次调用？ |
| **事中** | 上下文锚定 (Context Anchoring) | 上下文组装器 | 如何确保关键契约在上下文长度受限时不被挤出？ |
| **事后** | 生成校验 (Output Validation) | 静态分析 / 规则引擎 / 二次 AI 调用 | 如何低成本地检测生成结果是否违反全局约束？ |
| **对齐** | 差异修复 (Drift Correction) | 人工 / 自动化修复流程 | 当检测到漂移时，是修复局部还是回溯调整上层设计？ |

**分析要点**：
- 一致性保障不能只靠 Prompt，必须有**机制层**保障。
- 需要研究：不同保障机制的成本与检出率的权衡。

**可参考方案**：
- **ProjectGen**：采用 **LLM-as-Judge** 模式，JudgeA/JudgeS 从需求覆盖、接口一致性、依赖无环等维度打分，低于阈值则反馈迭代。
- **静态分析低成本校验**：类型检查、编译、linter 可快速捕获接口不匹配、未定义引用等错误。
- **SkillReducer 的 Faithfulness Checks**：验证压缩后的表示是否保留了原始技能的行为契约，可用于验证 L0-L1 摘要是否失真。
- **DeCC (2024)**：通过任务分解一致性比较，用独立 LLM 评估答案可靠性，可作为生成结果可信度评估方法。
- **AST 驱动的结构一致性校验**：
  - **STALL+ 的三阶段静态分析注入**：prompt 阶段提取 API 签名和有效符号集合；decoding 阶段用 AST 约束 token 概率（禁止生成未定义的标识符）；post-processing 阶段编译验证。
  - **AST diff 检测漂移**：将生成代码解析为 AST，与目标位置的预期 AST 结构进行 diff，检测是否引入了未声明的变量、错误的返回类型、缺失的异常处理等。
  - **LocalizeAgent (2025)**：通过 AST 提取 fan-in/fan-out、调用关系、变量使用、类耦合等轻量指标，生成自然语言摘要辅助 LLM 定位设计问题。

**建议分析方法**：
- 设计"一致性违规类型学"，分类统计各类违规的检出成本（静态分析/AST diff < 编译检查 < 单元测试 < 集成测试 < LLM-as-Judge）。
- 建立"漂移检测指标"：如生成代码与已有接口的依赖图 diff、命名规范匹配率、架构模式偏离度、AST 结构偏离度。

## 四、设计原则（Guiding Principles）

基于以上分析框架，后续方案设计应遵循以下原则：

### 4.1 最小惊奇原则 (Principle of Least Surprise)
AI 在任何一步都不应因为上下文中缺少"显然应该知道"的信息而做出违反常识的选择。这要求上下文组装器对人类常识与项目特定常识进行区分。

### 4.2 渐进展开原则 (Progressive Disclosure)
信息应按相关性逐步呈现，而非一次性堆叠。允许 AI 在需要时请求更多上下文（如果架构支持多轮交互）。

> 佐证：SkillReducer (2026) 的 progressive disclosure 机制在 600 个技能上验证了"按需加载"优于"全量灌输"。

### 4.3 焦点隔离原则 (Focus Isolation)
每一步任务应有且仅有一个明确的焦点。避免在单次调用中混合多个不相关的子任务。

### 4.4 契约锚定原则 (Contract Anchoring)
高层契约（接口、架构决策）应以**不可变引用**的形式嵌入低层任务的上下文中，而非每次复制全文。

> 佐证：CodeCompass (2026) 发现架构关键但语义疏远的文件易被忽略，必须通过显式图边锚定。

### 4.5 反馈收敛原则 (Feedback Convergence)
每一步生成后，应有机会将"实际发生了什么"反馈到全局状态，使后续步骤基于更准确的现实而非假设。

> 佐证：ProjectGen (2025) 的迭代优化机制，通过记录 Judge 反馈与版本差异，实现 ArchAgent 的记忆收敛。

### 4.6 有效窗口原则 (Effective Window Principle)
设计上下文预算时，以**MECW（最大有效上下文窗口）**而非标称 MCW 为约束。为不同任务类型分别测量 MECW。

> 佐证：Paulsen (2025) 发现 GPT-4.1 在复杂任务中 MECW 仅约 1k tokens，按 128k 设计必然失败。

### 方向 A：基于知识图谱 + AST 的上下文路由

### 方向 B：分层契约与摘要机制

- 摘要的更新触发条件是什么？如何保证摘要不失真？
- **参考**：ProjectGen 的 SSAT、用户实践的 `memory.md`、SkillReducer 的 faithfulness checks。

### 方向 C：动态任务分解策略
- 根据项目规模和模型能力，任务分解的粒度如何自适应调整？

- **参考**：Deep Agent Architecture 的 Orchestrator、Hierarchical Task Decomposition 模式。

### 方向 D：生成后校验与漂移检测
- 低成本校验：静态分析工具（类型检查、lint）能捕获多少一致性违规？
- **AST 结构校验**：将生成代码解析为 AST，对比预期结构（接口签名、异常声明、返回类型），检测结构性漂移。
- 高成本校验：用另一个 AI 调用做"架构审查"，其 ROI 如何？
- 校验失败后的修复策略：局部重写 vs 回溯上层 vs 人工接管。
- **参考**：ProjectGen 的 JudgeA/JudgeS、DeCC (2024)、SkillReducer 的 self-correcting feedback loop、STALL+ (2024)、LocalizeAgent (2025)。

### 方向 E：人类在环的最优介入点
- 在任务分解的哪个环节、上下文组装的哪个决策点，人类判断的价值最大？
- 如何设计"人类介入成本最小化"的交互界面？
- **参考**：Cursor 用户对自动上下文浅层的反馈、Deep Agent 的 Orchestrator 人工接管设计。

### 方向 F：上下文记忆与状态管理
- 跨会话的上下文如何持久化？哪些状态需要记住，哪些应该遗忘？
- 长项目中的"上下文衰减"问题：早期的决策在后期如何被记起？
- **参考**：ProjectGen 的 γ-摘要记忆机制、Anthropic 定义的 Context Engineering 学科。

## 附录 A：相关理论知识与关键概念

> 本附录整理与框架密切相关的理论基础，供后续深入研究时参考。

### A.1 最大有效上下文窗口 (Maximum Effective Context Window, MECW)
- **定义**：模型在特定任务类型上保持可接受性能的最大输入长度，区别于标称的架构上限 (MCW)。
- **测量方法**：将特定事实嵌入上下文不同位置，测试模型能否正确召回；或观察准确率-长度曲线的拐点。
- **关键发现**：MECW 随任务复杂度急剧下降。简单检索可达数万 token，复杂多步推理可能仅数百 token。
- **来源**：Paulsen (2025), NoLiMa Benchmark (LMU Munich & Adobe Research, ICML 2025)。

### A.2 迷失在中间 (Lost in the Middle)
- **定义**：LLM 倾向于忽略位于长上下文中间位置的信息，对开头和结尾的信息回忆更可靠。
- **成因**：位置编码偏见 + 注意力机制的 U 型分布。
- **缓解策略**：关键信息重排序（放首尾）、提示压缩、段落级重新排序（按注意力分数）、锚点提示 (anchor prompting)。
- **来源**：Liu et al. (2024), Hutter et al. (2025)。

### A.3 注意力稀释 (Attention Dilution)
- **定义**：随着上下文增长，Transformer softmax 归一化导致每 token 的注意力权重被稀释，模型难以聚焦任何单一信息。
- **后果**：Context Rot（上下文腐化）—— 输出质量随上下文增长而系统性下降。
- **缓解策略**：子 Agent 架构隔离搜索过程、上下文压缩、RAG 替代全量加载。
- **来源**：Anthropic Multi-Agent System 研究 (2025-2026), MorphLLM Context Rot 分析。

### A.4 上下文工程 (Context Engineering)
- **定义**（Anthropic）：在推理过程中策划最优 token 集合的学科。不同于 Prompt Engineering（关注单一输入），Context Engineering 管理多轮交互中的整体上下文状态。
- **目标**：找到任务所需的最小高信号 token 集合。
- **实践**：上下文隔离、子 Agent 架构、检索增强生成 (RAG)、记忆树构建。

### A.5 渐进展开 (Progressive Disclosure)
- **定义**：将信息分层组织，仅在需要时展示详细内容，避免一次性信息过载。
- **应用**：SkillReducer 的技能体重构、分层 RAG、多级摘要。
- **验证**：SkillReducer 在 600 技能上证明压缩后（更少 token）的功能质量反而提升。

### A.6 导航悖论 (Navigation Paradox)
- **定义**：CodeCompass (2026) 提出——更大上下文窗口不会消除结构导航需求，只是将失败模式从"检索容量不足"转移到"导航显著性不足"。
- **启示**：代码理解不仅需要检索能力，更需要**显式结构导航**（调用图、继承图、依赖图）。

### A.7 LLM-as-Judge
- **定义**：使用独立的 LLM 调用对生成结果进行多维度评估（如需求覆盖、接口一致性、依赖合理性）。
- **应用**：ProjectGen 的 JudgeA（架构评估）和 JudgeS（骨架评估）。
- **局限**：增加成本、可能引入评审偏见。适合关键节点（架构锁定、集成前），不适合每步都调用。

### A.8 AST 作为结构化中间表示（Structured Intermediate Representation）
- **定义**：AST（抽象语法树）是源代码的树形结构化表示，每个节点对应声明、语句、表达式或类型定义等语法构造。
- **为什么 AST 适合上下文控制**：
  - **语法完整性**：AST 节点天然具有语法边界，按 AST 裁剪不会生成"半个函数"或断裂的语句块（避免 Frankenstein context）。
  - **精确提取**：可精确提取函数签名、类成员、import 声明、访问修饰符等，而无需加载完整方法体。
  - **多粒度表示**：同一代码可通过 AST 生成不同粒度的表示——完整源码、仅签名、SBT 序列化 AST、简化 AST（删除冗余节点）、控制流摘要。
  - **结构到文本的桥梁**：AST 可将代码结构转化为自然语言摘要（如"该函数包含一个 try-except 块，在异常时返回空列表"），实现代码与设计文档的语义对齐。
- **关键操作**：
  - **AST Pruning（剪枝）**：删除与当前任务无关的子树（如目标函数未调用的私有方法）。
  - **AST Slicing（切片）**：基于数据流/控制流保留影响特定变量的最小语句集（Weiser 反向切片）。
  - **AST Serialization（序列化）**：将 AST 转化为 LLM 可读的线性序列（如 SBT、pre-order traversal），在保留结构信息的同时减少 token 数。AST(NIT) (2026) 证明序列化 AST 比原始代码更短且摘要质量相当。
  - **AST-based Chunking（分块）**：cAST (EMNLP 2025) 证明基于 AST 边界的分块比行级分块在检索和生成任务上均更优。
- **来源**：Su & McMillan (2026), HASTE (2026), cAST (EMNLP 2025), AstraAI (2026), AST(NIT) (2026), SimAST-GCN (2024)。
