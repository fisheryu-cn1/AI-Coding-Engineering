# Wiki 与图数据库混合架构策略笔记

> 基于「轻量级文本知识管理」与「AI 代码生成上下文供给」场景的架构讨论整理  
> 核心问题：多大规模下 Wiki 达到瓶颈？如何以最小成本引入图数据库能力？

---

## 一、核心结论

1. **Wiki 的链接机制本质上就是一张图**，在规模不大（< 150 个实体）且以人工浏览为主的场景下，可覆盖图数据库约 70% 的日常价值。
2. **AI 代码生成场景改变了平衡点**：AI 需要的是「可计算的图」而非「可读的图」，对关系遍历深度、上下文噪音容忍度、实时性要求更高，因此图数据库的优势会**提前显现**。
3. **最优解不是二选一，而是分域混合**：将「结构化域」（API、ER 模型、模块依赖）放入图数据库，将「叙事域」（需求、架构设计、ADR）保留在 Wiki，通过桥梁层统一为 AI 和人类提供服务。

---

## 二、Wiki vs 图数据库：效率平衡点估算

### 2.1 一般文本知识场景

| 规模 | 实体/文档数 | Wiki 表现 | 图数据库表现 | 建议 |
|------|-----------|---------|------------|------|
| **小型** | < 150 | ✅ 完美。全文搜索+链接足够 | 过度设计，ROI 为负 | **只用 Wiki** |
| **中型** | 150–800 | ⚠️ 开始出现信息孤岛，跨主题查询需多次搜索 | 开始显现优势 | **Wiki 为主，关键关系用图** |
| **临界点** | **800–1500** | ❌ 噪音显著，3 跳以上关系不可行 | ✅ 多跳查询稳定 | **评估迁移** |
| **大型** | > 1500 | ❌ 维护困难，上下文经常遗漏 | ✅ 唯一能可靠处理复杂依赖的方案 | **必须上图数据库** |

### 2.2 AI 代码生成场景（平衡点前置）

AI 场景对上下文组装的要求更高，因此临界点更低：

- **文档 < 150，团队 < 5 人**：Markdown + 文件夹足够。
- **文档 150–800，或微服务 3–8 个**：增强 Wiki（如 Obsidian + Dataview），但需开始关注跨模块关系。
- **文档 > 800，或微服务 > 5 个，或 AI 频繁因「跨模块影响遗漏」而生成错误代码**：**必须引入图数据库**。

### 2.3 更精确的判定指标

比起数文档数量，以下指标更能反映真实瓶颈：

| 指标 | 阈值 | 含义 |
|------|------|------|
| **关系密度** | 有效链接数 / 文档总数 > 6 | 关系复杂，图数据库能显著提升 AI 上下文质量 |
| **上下文断裂频率** | 连续 3–5 次 AI 因遗漏跨模块信息而出错 | Wiki 的上下文组装能力已达极限 |
| **平均组装跳数** | ≥ 3 跳 | AI 需要理解「函数 → 模块 → 架构决策 → 依赖服务 API」，Wiki 难以支撑 |

## 三、分域混合策略：结构化域 vs 叙事域

### 3.1 划分逻辑

| 维度 | 图数据库（结构化域） | Wiki（叙事域） |
|------|-------------------|---------------|
| **内容本质** | API 契约、ER 关系、模块依赖、配置 Schema | 需求背景、设计决策、ADR、最佳实践、会议纪要 |
| **主要消费者** | AI 工具（代码生成、影响分析）、CI/CD | 人类开发者、产品经理、架构师 |
| **结构强度** | 强 Schema，关系密集 | 弱 Schema，自由格式 |
| **变更模式** | 中频，跟随版本迭代 | 高频，日常演进 |
| **查询方式** | 多跳遍历、模式匹配、子图提取 | 全文搜索、线性阅读、时间线回溯 |

> **核心洞察**：API 文档和 ER 模型是「代码世界的地图」，适合图结构；需求说明和架构设计是「决策过程的叙事」，适合 Wiki。

### 3.2 这种划分如何改变平衡点

采用分域混合后，图数据库只需管理**核心实体的 15–25%**，却能覆盖 **80% 的 AI 结构化查询需求**。这意味着：

- **200 页文档、3–5 个微服务、或 50+ API 端点**即可启动混合架构。
- 无需等待整体规模达到纯图数据库方案的临界点。

## 四、双域一桥架构设计

### 4.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                      人类入口层（Wiki）                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
│  │   需求说明    │  │   架构设计    │  │   ADR / 最佳实践  │  │
│  │  (叙事域)    │  │  (叙事域)    │  │    (叙事域)      │  │
│  └──────┬───────┘  └──────┬───────┘  └────────┬─────────┘  │
│         │                 │                    │            │
│         └─────────────────┼────────────────────┘            │
│                           │ 双向链接（持久化 URI 锚点）        │
│                      ┌────┴────┐                          │
│                      │  桥梁层  │  ← GraphQL / REST 网关    │
│                      │ (统一 ID │    实体名解析服务           │
│                      │  映射表) │                          │
│                      └────┬────┘                          │
│                           │ 自动同步 / 按需查询              │
│         ┌─────────────────┼────────────────────┐            │
│         │                 │                    │            │
│  ┌──────▼───────┐  ┌─────▼──────┐  ┌──────────▼────────┐  │
│  │   API 文档    │  │   ER 模型   │  │   模块依赖网络     │  │
│  │  (结构域)    │  │  (结构域)   │  │    (结构域)      │  │
│  └──────────────┘  └────────────┘  └───────────────────┘  │
│              图数据库（Neo4j / Kùzu / ArangoDB）            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
              ┌───────────────────────────────┐
              │   AI Context Assembly Service  │
              │   (RAG Pipeline / Agent 层)    │
              │                                │
              │  1. 接收代码任务               │
              │  2. 图数据库 → 提取子图        │
              │  3. Wiki → 检索决策背景        │
              │  4. 融合 → 注入 LLM 上下文     │
              └───────────────────────────────┘
```

### 4.2 关键机制

#### （1）跨域链接：持久化 URI 锚点

- 图数据库中每个实体拥有持久 URI：`graph://service-user/entity/UserAPI`
- Wiki 中引用时使用标准化链接：`[UserAPI](graph://service-user/entity/UserAPI)`
- 图数据库实体属性中存储对应 Wiki 页面 URL：`wiki_url: "https://wiki.company.com/arch/user-service"`

**目的**：即使实体改名，URI 不变，跨域链接不断裂。

#### （2）同步策略

| 同步方式 | 适用场景 | 实现 |
|---------|---------|------|
| **实时查询** | AI 代码生成（低延迟要求） | Context Service 同时查询两系统，500ms 内组装 |
| **预计算索引** | 复杂影响分析 | 夜间任务将 Wiki 中的实体提及预关联到图节点 |
| **事件驱动** | API 变更触发文档更新 | 代码合并后，CI 更新图数据库，并通知 Wiki 相关页面 |

#### （3）AI 上下文融合模板

AI 不应简单拼接两种数据，而应按以下结构融合：

```markdown
[结构化上下文 - 来自图数据库]
---
模块：UserService
依赖：AuthService(v2.1), NotificationService
数据库表：users, user_profiles
API 端点：POST /api/v2/users, GET /api/v2/users/{id}
影响范围：直接下游 3 个服务，间接下游 7 个服务
---

[叙事上下文 - 来自 Wiki]
---
设计决策（ADR-042）：
- UserService 在 2024-Q3 从单体拆分，采用 CQRS 模式
- 用户创建流程需要异步发送欢迎邮件
- 历史遗留：{id} 参数兼容旧版 UUID 和 新版 ULID

需求背景（REQ-2024-088）：
- 需要支持企业客户的 SSO 登录，因此 AuthService 依赖不可绕过
---

[当前任务]
实现用户批量导入功能...
```

**原则**：图数据提供「精确的技术约束」，Wiki 提供「意图和业务逻辑」。

## 五、最小可行方案（MVP）

如果今天就想落地，可采用以下轻量组合：

| 角色 | 工具建议 | 理由 |
|------|---------|------|
| **Wiki** | Obsidian / Outline / MkDocs | 支持双向链接，可导出标准化锚点 |
| **图数据库** | **Kùzu**（嵌入式）或 **DuckDB + 网络表** | 无需独立服务，可嵌入项目 |
| **桥梁** | Python 脚本（约 50 行） | 解析 Wiki 的 `[[...]]` 链接，提取 `graph://` URI |
| **AI 集成** | Cursor Rules / Kimi CLI 上下文钩子 | 在 AI 请求前调用脚本组装上下文 |

### MVP 工作流

1. **在 Wiki 中引用结构化实体**：
   ```markdown
   用户创建流程调用 [CreateUser](graph://user-api/CreateUser)，
   该接口在 ADR-042 中定义为异步模式。
   ```

2. **将 OpenAPI/Swagger 导入 Kùzu**：
   - 构建 API 调用关系图、服务依赖图。

3. **编写 Context Assembler**（伪代码）：
   ```python
   def assemble_context(code_file_path):
       # 1. 从代码文件提取涉及的实体（函数、类、API 名）
       entities = extract_entities(code_file_path)
       
       # 2. 从图数据库提取子图（依赖、调用链）
       subgraph = graphdb.query(f"""
           MATCH (n)-[*1..3]-(m)
           WHERE n.name IN {entities}
           RETURN n, relationships, m
       """)
       
       # 3. 从 Wiki 检索提及这些实体的段落
       narrative = wiki.search(entities, limit=5)
       
       # 4. 融合并返回
       return fuse_into_prompt(subgraph, narrative)
   ```

## 六、潜在陷阱与应对

| 陷阱 | 表现 | 应对 |
|------|------|------|
| **命名漂移** | Wiki 里叫「用户服务」，图数据库里叫 `UserService`，AI 无法关联 | 建立**同义词表**或强制使用 URI 锚点 |
| **单向同步** | Wiki 更新了设计，但图数据库里的 API 契约还是旧的 | 设计文档中明确标注**版本标签**，AI 组装时做版本校验 |
| **过度工程** | 一开始就把所有 Wiki 内容都试图结构化 | 坚持「结构域只放机器必须先理解的关系」，其余留在叙事域 |
| **心智模型分裂** | 团队成员不知道信息该写在哪 | 制定清晰约定：「为什么」在 Wiki，「是什么」在图数据库 |

## 七、决策清单

在决定是否采用混合架构时，可逐项勾选：

- [ ] 项目是否包含 3 个以上微服务或 50 个以上 API 端点？
- [ ] AI 是否曾因「未感知跨模块影响」而生成错误代码？
- [ ] 是否存在大量需要机器推理的关系密集型知识（ER 模型、调用链、依赖网）？
- [ ] 人类是否仍需要频繁阅读和编辑需求说明、架构决策文档？
- [ ] 团队是否愿意为 15–25% 的核心实体维护一个结构化图谱？

**如果勾选 3 项以上，混合架构大概率是你的最优解。**
