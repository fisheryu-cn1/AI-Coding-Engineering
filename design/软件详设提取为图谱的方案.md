把软件开发详细设计方案自动整理成知识图谱，目前最成熟的方案是LLM 驱动的领域自适应抽取流水线。核心思路不是让 LLM "读文档写总结"，而是让模型按照预定义的软件工程本体（Ontology）去结构化提取实体和关系，最终形成可查询、可推理的图数据库。

---

一、整体架构：四层流水线

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  文档预处理  │ → │  语义抽取    │ → │  知识融合    │ → │  图库存储    │
│  (Parsing)  │    │ (Extraction)│    │ (Fusion)    │    │ (Storage)   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

第 1 层：文档预处理

详细设计文档通常是 Word、PDF 或 Markdown，需要先解析为结构化文本：

文档格式	推荐工具	关键处理	
PDF	LlamaParse / Marker / unstructured.io	保留章节层级、表格、代码块	
Word (.docx)	python-docx / mammoth	提取标题层级（Heading 1-6）作为文档骨架	
Markdown	原生解析	直接利用标题结构和代码块	

关键技巧：不要直接把所有文本扔给 LLM。先按文档结构（章节标题）切分，保留层级关系。例如：

```
1. 系统架构设计
   1.1 模块划分
       1.1.1 用户管理模块
       1.1.2 订单处理模块
   1.2 接口定义
2. 数据库设计
   2.1 表结构
```

第 2 层：语义抽取（核心）

这是整个方案的关键。不要直接用通用 SPO 抽取，而要定义软件工程专用本体（Ontology）。

推荐的实体类型（节点）

实体类型	说明	示例	
`Module`	功能模块	用户管理模块、支付网关	
`Class`	类/组件	`UserService`, `OrderController`	
`Interface`	接口定义	`IUserRepository`, REST API	
`Function`	函数/方法	`authenticate()`, `createOrder()`	
`DatabaseTable`	数据库表	`t_user`, `t_order`	
`Field`	字段/属性	`userId`, `orderStatus`	
`DataFlow`	数据流	用户注册请求流、订单状态变更流	
`Technology`	技术栈	MySQL, Redis, Kafka, Spring Boot	
`DesignPattern`	设计模式	策略模式、工厂模式	

推荐的关系类型（边）

关系类型	说明	
`DEPENDS_ON`	模块/类之间的依赖	
`CALLS`	函数调用关系	
`IMPLEMENTS`	实现接口	
`INHERITS_FROM`	继承关系	
`CONTAINS`	包含关系（模块包含类）	
`READS_FROM`	从数据库表读取	
`WRITES_TO`	写入数据库表	
`USES_TECHNOLOGY`	使用某技术	
`INPUT_OF` / `OUTPUT_OF`	数据流的输入输出	
`DESIGNED_BY`	设计文档归属	

抽取 Prompt 设计（关键）

对每个文档块，使用类似这样的结构化 Prompt：

```python
SYSTEM = """你是一个软件工程知识抽取专家。请从以下详细设计文档片段中，提取结构化的知识图谱三元组。
只提取与软件设计相关的实体和关系，忽略寒暄、背景介绍等无关内容。

实体类型限定为：Module, Class, Interface, Function, DatabaseTable, Field, DataFlow, Technology, DesignPattern
关系类型限定为：DEPENDS_ON, CALLS, IMPLEMENTS, INHERITS_FROM, CONTAINS, READS_FROM, WRITES_TO, USES_TECHNOLOGY, INPUT_OF, OUTPUT_OF

输出格式：JSON数组，每个元素为 {"subject": "...", "relation": "...", "object": "...", "subject_type": "...", "object_type": "..."}
"""

USER = f"""文档片段：
{chunk_text}

请提取所有三元组，确保：
1. 实体名称使用文档中的准确术语
2. 关系方向正确（A CALLS B 表示 A 调用 B）
3. 包含实体类型标注
"""
```

第 3 层：知识融合（去重与消歧）

抽取后的原始三元组会有大量重复和歧义，必须经过融合：

问题	解决方案	
同名异义	`User` 在 A 文档指表，在 B 文档指类 → 根据上下文和类型标注区分	
异名同义	`用户表` / `t_user` / `user_table` → 使用 LLM 做实体链接（Entity Linking）	
关系冲突	A 说 `Module X DEPENDS_ON Module Y`，B 说无关 → 按出现频次或文档权威性仲裁	
属性补全	合并同一实体的属性（如 `UserService` 在不同文档中描述的职责拼接）	

实现建议：使用嵌入模型（如 `bge-large-zh`）计算实体名称的语义相似度，高于阈值则视为同一实体，再让 LLM 做最终确认。

第 4 层：图库存储与可视化

图数据库	特点	适用场景	
Neo4j	生态最成熟，Cypher 查询直观，有 LLM Knowledge Graph Builder 插件	大多数场景首选	
NebulaGraph	分布式，适合超大规模	企业级、海量文档	
TigerGraph	高性能分析	复杂图算法、实时分析

二、针对软件设计文档的增强策略

策略 1：Schema 先验约束（最重要）

不要完全让 LLM "自由发挥"实体类型。先定义好软件工程的 Graph Schema，作为抽取约束：

```json
{
  "nodes": {
    "Module": {"properties": ["name", "responsibility", "doc_source"]},
    "Class": {"properties": ["name", "language", "package", "doc_source"]},
    "Interface": {"properties": ["name", "protocol", "endpoint", "doc_source"]}
  },
  "edges": {
    "DEPENDS_ON": {"from": ["Module", "Class"], "to": ["Module", "Class"]},
    "CALLS": {"from": ["Function", "Class"], "to": ["Function", "Class"]}
  }
}
```

腾讯云 GraphRAG 的做法是提供预置模板 + AI 一键扩展，你可以借鉴：先手动定义核心 Schema，再让 LLM 基于文档内容自动扩展补充。

策略 2：文档结构感知切分

详细设计文档通常有固定模板（如：模块设计 → 接口设计 → 数据库设计 → 异常处理）。切分时保留章节路径作为元数据：

```python
chunk_metadata = {
    "doc_title": "订单系统详细设计",
    "section_path": "3.2.1 订单创建接口",
    "section_type": "interface_design",  # 可指导抽取策略
    "page_range": "45-48"
}
```

这样抽取出的三元组可以回溯到原文，也方便后续做文档-图谱双向同步（Living Documentation）。

策略 3：代码片段特殊处理

设计文档中常包含伪代码或 SQL 建表语句。对这些内容用专门的解析器（而非通用 LLM）提取：

- SQL DDL → 提取 `DatabaseTable`, `Field`, 关系 `CONTAINS`
- 类图/伪代码 → 提取 `Class`, `Function`, 关系 `CALLS`, `INHERITS_FROM`

三、推荐技术栈（可直接落地）

环节	推荐工具	说明	
文档解析	LlamaParse / Marker / unstructured	PDF/Word 转结构化文本	
文本切分	LangChain / LlamaIndex	语义切分 + 结构切分混合	
实体/关系抽取	Claude Sonnet / GPT-4o / GLM-4.5	用结构化输出（JSON mode / Pydantic）强制格式	
嵌入模型	bge-large-zh-v1.5 / m3e	实体消歧、相似度计算	
图数据库	Neo4j Community / AuraDB	有成熟的 Python 驱动和可视化	
流水线编排	LangChain Graph Transformer / 自研脚本	Neo4j 官方提供 `llm-graph-builder`	
可视化	Neo4j Browser / yFiles / D3.js	交互式探索

四、一个最小可运行的示例

以下是用 Python + Neo4j + OpenAI 的最简流水线：

```python
from neo4j import GraphDatabase
import openai
import json

# 1. 文档切分（示例）
chunks = [
    {"text": "用户管理模块包含 UserService 类和 UserRepository 接口。UserService 调用 UserRepository 查询用户数据。", "source": "doc1_sec3.1"},
    {"text": "订单模块 OrderService 依赖用户模块的 UserService 获取用户信息。", "source": "doc1_sec4.2"}
]

# 2. LLM 抽取
def extract_triples(text):
    response = openai.chat.completions.create(
        model="gpt-4o",
        messages=[{
            "role": "system",
            "content": "提取软件设计文档中的实体和关系，输出JSON数组：[{subject, relation, object, subject_type, object_type}]"
        }, {
            "role": "user",
            "content": text
        }],
        response_format={"type": "json_object"}
    )
    return json.loads(response.choices[0].message.content)["triples"]

# 3. 写入 Neo4j
driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))

def add_triples(tx, triples):
    for t in triples:
        tx.run("""
            MERGE (s:Entity {name: $subject, type: $subject_type})
            MERGE (o:Entity {name: $object, type: $object_type})
            MERGE (s)-[r:RELATION {type: $relation}]->(o)
        """, **t)

with driver.session() as session:
    for chunk in chunks:
        triples = extract_triples(chunk["text"])
        session.execute_write(add_triples, triples)
```

五、进阶方向

1. GraphRAG 查询：把知识图谱和向量检索结合，支持"找出所有依赖订单模块且使用 MySQL 的接口"这类多跳推理查询 
2. 文档-图谱同步：当设计文档更新时，自动 diff 并增量更新图谱（Living Documentation）
3. 跨文档关联：多个系统的详细设计文档抽取后，在图谱层做跨系统依赖分析
4. 可视化探索：用 Neo4j Bloom 或自研前端做交互式架构图生成

---

总结：最佳方案是"Schema 先验 + LLM 抽取 + 嵌入消歧 + 图库存储" 的四层流水线。不要追求一步到位，先定义好软件工程的本体，再让 LLM 做有约束的抽取，最后通过 Neo4j 落地。这样生成的知识图谱不仅能查，还能支撑架构分析、影响面评估等高级应用。
