### 2. 上下文工程（`context-engineering/`）

这是近期研究最集中的方向，围绕"如何为 AI Coding Agent 提供高质量、可控、低成本的上下文"。

| 文件 | 核心问题 | 关键结论 |
|---|---|---|
| `知识图谱管理Agent上下文.md` | 能否用知识图谱管理设计文档/API 文档，并在代码生成每步动态提取上下文 | 可行且已有大量支撑；GraphRAG/TREEFRAG 可将上下文压缩比提升至 18:1–24:1，保持 94%+ 准确率；关键在于细粒度 schema、步骤感知检索、生成器感知压缩 |
| `API与框架文档自动化知识图谱提取.md` | 如何将 API/框架文档自动提取为知识图谱以服务代码生成 Agent | 提出分层混合提取架构：DOM 解析显式结构 + LLM 语义增强 + 图数据库存储；设计 API 卡片与 Grounding 三层溯源模型 |
| `API信息提供策略.md` | 生成代码时应提供完整 API 说明还是仅引用 URL | 分层渐进式提供最优：先注入 API 概要卡片，模型按需通过工具拉取详细文档；过早/过度检索会引入噪声 |
| `代码图谱与LLM上下文管理工具调研.md` | 开源代码图谱/上下文打包工具（RepoGraph、Joern、Sourcegraph、Aider repo-map、Repomix 等）的可用性与可行性 | 公开工具多在特定场景有效，但在数十万行级私有仓库上的自动编码仍需在开源基础上做深度定制或自研 |
| `Context7与代码图谱结合设计分析.md` | Context7 的设计思路能否与代码图谱结合用于私有代码库 | Context7 的"文档上下文服务"思路可迁移到私有库：将代码图谱 + 设计文档图谱统一为 MCP 可查询的上下文服务 |

**主题内关联**：

```
知识图谱管理上下文（顶层设计）
    ├── API/框架文档知识图谱化（文档侧上下文来源）
    ├── API 信息提供策略（上下文消费协议）
    ├── 代码图谱工具调研（代码侧上下文来源）
    └── Context7 + 代码图谱（上下文服务化方案）
```

**后续方向**：
- 在真实 Spring/微服务项目中验证"API 卡片 + 代码图谱 + MCP 工具"的上下文方案；
- 设计统一的上下文 schema，兼容设计文档、API 文档、代码结构三类来源；
- 建立上下文质量评估指标（压缩比、召回率、任务成功率、token 成本）。

## 近期研究主线

近期研究可归纳为一条清晰主线：

> **从"模型能力"转向"上下文能力"，从"提示技巧"转向"知识基础设施"。**

具体而言：
1. **问题诊断**：LLM 长上下文能力存在根本极限（见 `references/ContextEngineering/` 中 A 组论文）。
2. **方案探索**：用知识图谱、代码图谱、API 文档图谱组织项目知识，替代简单的文本堆叠。
3. **接口设计**：通过 MCP、SDD 规范、API 卡片等机制，让 Agent 按需、可控地消费上下文。
4. **应用落地**：针对 Spring 等传统架构，设计显式化、规范化的 AI 自动化演进路径。

---

## 后续优先研究方向建议

1. **统一上下文 schema 设计**
   - 整合设计文档、API 文档、代码结构三类知识，定义可机器消费的图模型。

2. **MCP 上下文服务原型**
   - 基于代码图谱 + 文档图谱，实现一个可向 Claude Code/Cursor 等工具提供结构化查询的 MCP 服务器。

3. **SDD 与上下文工程融合**
   - 让规范（Spec）成为上下文的"一级公民"，代码生成过程严格锚定规范。

4. **Spring/微服务项目的 AI 友好型改造**
   - 将隐式架构显式化，建立领域规范，验证 AI 自动化重构/迭代的效果。

5. **上下文质量评估体系**
   - 建立压缩比、召回率、任务成功率、token 成本等多维度指标。

6. **本体论驱动的领域建模**
   - 在目标业务领域中构建本体，验证 Ontology-Grounded RAG 对代码生成的增益。
