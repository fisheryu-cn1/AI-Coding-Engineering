## 二、分文献详细阅读笔记

### A. 长上下文能力的极限

> 注：A 组 6 篇文献的详细笔记与 2026-05-16 初版保持一致，核心结论可概括为：**LLM 厂商宣称的上下文窗口是架构上限，而非有效工作记忆；实际可稳定利用的上下文通常只有宣称值的 30%–60%，且存在任务依赖性、位置偏置与临界阈值**。以下仅保留精炼版要点，详见初版报告。

#### 1. Context Is What You Need（Paulsen, 2025）
- **核心命题**：最大上下文窗口（MCW）与最大有效上下文窗口（MECW）存在巨大鸿沟，且 MECW 是任务相关的动态变量。
- **关键发现**：多数顶尖模型在 1,000 tokens 左右即出现严重精度退化；RAG 在 MECW 内可将准确率提升至近 100%，但超过 MECW 后幻觉率飙升；Agentic 框架中存在级联失效。
- **实践启示**：应将任务拆分为子 Agent，使每个 Agent 的上下文控制在 MECW 以内。

#### 2. Lost in the Middle（Liu et al., 2023）
- **核心命题**：模型对相关信息位置高度敏感，扩展上下文窗口不等于更好利用上下文。
- **关键发现**：U 型性能曲线——开头和结尾性能最高，中间显著下降；decoder-only 模型存在首因与近因双重偏置；开放域 QA 中检索文档数从 20 增至 50 仅带来约 1–1.5% 提升。
- **实践启示**：RAG 应重排序或截断文档，将最相关信息置于上下文前部。

#### 3. Intelligence Degradation in Long-Context LLMs（Wang et al., 2026）
- **核心命题**：开源模型处理长上下文时存在精确临界退化阈值，超过后性能断崖式下跌。
- **关键发现**：Qwen2.5-7B（128K 窗口）临界阈值约为最大长度的 **43.2%**（约 55K tokens），退化区 F1 从 0.556 跌至 0.302。
- **实践启示**：为保持稳定，上下文应控制在最大容量的 40% 以下（约 51K tokens）。

#### 4. Limitations of Normalization in Attention Mechanism（Mudarisov et al., 2025）
- **核心命题**：Softmax 归一化随上下文长度增长导致注意力权重趋向 1/L，难以区分相关与无关 token。
- **关键发现**：长度无关归一化的注意力权重必然满足 C₁/L ≤ αᵢ ≤ C₂/L；仅约 0.06L（<6%）的 token 被选中后，经验分布即与均匀分布无统计差异。
- **实践启示**：保持激活集较小，监控注意力熵，优先使用稀疏归一化器或长度感知缩放。

#### 5. On the Fundamental Limits of LLMs at Scale（Mohsin et al., 2026）
- **核心命题**：LLM 规模化收益受幻觉、上下文压缩、推理退化、检索脆弱性、多模态错位五大根本极限约束。
- **关键发现**（上下文相关）：位置欠训练、位置编码饱和、Softmax 拥挤共同导致有效上下文随名义长度亚线性增长；Llama3.1 70B 128K 窗口有效利用仅约 64K。
- **实践启示**：有效上下文是硬性上限，RAG 不是万能解；应采用迭代式 planner-reader 架构。

#### 6. Context Rot（Hong et al., Chroma, 2025）
- **核心命题**：18 个主流 LLM 在长上下文场景下均表现出 Context Rot——性能随输入 token 增加而系统性腐烂。
- **关键发现**：几乎所有模型都受长度影响；打乱句序的 haystack 反而优于逻辑连贯原文；LongMemEval 中完整输入（~113K）显著低于聚焦输入（~300 tokens）。
- **实践启示**：上下文工程的核心是“信息存在不等于信息可用”，必须通过预处理将单次请求上下文压缩到有效窗口内。

### B. 上下文工程的基础设施化

#### 7. On the Impact of AGENTS.md Files on the Efficiency of AI Coding Agents（Lulla et al., 2026）

**核心命题**：仓库根目录的 `AGENTS.md` 文件能否让 AI 编程代理更快、更省 token 地完成真实 PR 任务？

**方法论亮点**：
- 使用 OpenAI Codex（gpt-5.2-codex），在 10 个仓库、124 个已合并 GitHub PR 上进行配对实验。
- 同一任务、同一仓库快照，分别在有/无 `AGENTS.md` 条件下运行。
- 控制 PR 规模：≤100 行改动、≤5 个文件、纯代码改动、且在仓库引入 `AGENTS.md` 之后。

**关键发现**：
| 指标 | 无 AGENTS.md | 有 AGENTS.md | 变化 |
|---|---|---|---|
| 中位完成时间 | 98.57s | 70.34s | **降低 28.64%** |
| 平均完成时间 | 162.94s | 129.91s | 降低 20.27% |
| 平均输出 token | 5,744.81 | 4,591.46 | 降低 20.08% |
| 中位输出 token | 2,925 | 2,440 | 降低 16.58% |

- 完成时间和输出 token 的下降在统计上显著（Wilcoxon p<0.05）。
- 输入 token 平均降低约 9.73%，但中位数基本持平。
- 50 个任务的 sanity check 确认：代理输出不是空改或胡乱改，质量行为基本可比。

**实践启示**：
- `AGENTS.md` 这类仓库级持久化上下文文件有**可测量的效率收益**。
- 原因推测：预先说明项目结构、构建命令、编码规范，减少了代理的探索性导航和反复试错。
- 局限：仅测了 OpenAI Codex；仅选小范围 PR；未做完整正确性评估。

#### 8. Codified Context: Infrastructure for AI Agents in a Complex Codebase（Vasilopoulos, 2026）

**核心命题**：单文件 manifest（`.cursorrules`、`CLAUDE.md`、`AGENTS.md`）在小项目里够用，但当代码库涨到 10 万行时，代理会“失忆”、重复犯错、失去一致性。如何把项目知识规模化地组织成基础设施？

**研究场景**：
- 作者用 Claude Code 作为唯一代码生成工具，在 70 天（兼职）里开发了一个 **108,000 行 C# 的实时分布式多人游戏模拟系统**。
- 作者本人主要背景是化学而非软件工程，因此这也是一个“领域专家借助 AI 跨专业开发”的测试案例。

**三层架构（Codified Context Infrastructure）**：

```
┌─────────────────────────────────────────┐
│  Tier 1: Constitution（热内存，始终加载）   │  ~660 行
│  编码规范、命名约定、构建命令、触发表、已知失败模式 │
├─────────────────────────────────────────┤
│  Tier 2: Specialized Agents（按任务调用）  │  19 个 agent，~9,300 行
│  领域专家代理：网络协议设计师、坐标巫师、UI 同步专家等 │
├─────────────────────────────────────────┤
│  Tier 3: Knowledge Base（冷内存，按需检索） │  34 个文档，~16,250 行
│  子系统规范文档，通过 MCP 检索服务查询         │
└─────────────────────────────────────────┘
```

**关键数据**：
- 283 个开发会话，2,801 个人类提示，1,197 次代理调用，16,522 个代理回合。
- 上下文基础设施占代码量的 **24.2%**（26,200 行 / 108,256 行代码）。
- 结构化会话约占 13%，ad-hoc 会话约占 87%。
- 超过 80% 的人类提示在 100 词以内。
- 1,478 次 MCP 检索调用，194 个代理会话明确读取了知识库文档。

**四个案例研究**：
| 案例 | 机制 | 效果 |
|---|---|---|
| Save System | 作为跨会话协调文档 | 规范在 74 个会话、12 个代理对话中被引用，后续 5 个涉及持久化的功能都没引入 bug |
| UI Sync Routing | 捕获经验 | 商店同步经验写成规范后，后续 10+ 会话、25+ 代理对话直接复用 |
| Drop System | 知识缺口检测 | 重构前检索发现缺少文档，先补文档再重构，避免盲目修改 |
| Deterministic RNG | 嵌入领域知识协同调试 | 网络协议设计师代理靠规范预载的确定性理论，找出 5 次尝试都没解决的同步 bug |

**实践指南（六条）**：
1. 基础宪法就能起大作用
2. 让规划代理先收集上下文
3. 自动路由，否则人会忘
4. 一件事解释两次就写下来
5. 卡住就创建一个专家代理并重启会话
6. 陈旧的规范会误导代理

**局限**：单开发者、单项目、用 Claude Code，缺乏对照实验；实时分布式模拟游戏比一般应用需要更多文档。

#### 9. Knowledge Activation: AI Skills as the Institutional Knowledge Primitive for Agentic Software Development（Bakal, 2026）

**核心命题**：企业软件组织积累了大量关键制度性知识（架构决策、部署流程、合规政策、事故 playbook），但这些知识仍困在“为人类阅读设计”的格式中。有效 agentic 软件开发的瓶颈不是模型能力，而是**知识架构**。

**核心概念**：
- **Institutional Impedance Mismatch**：模型参数知识与组织制度性知识之间的结构性错位。
- **Context Window Economy**：上下文窗口是一个有限容量的通信信道，知识交付受 token 预算、注意力衰减和延迟成本三重约束。
- **Knowledge Activation**：将潜在组织知识通过**编码（codification）→ 压缩（compression）→ 注入（injection）**三阶段转化为代理可执行形式。
- **Atomic Knowledge Unit（AKU）**：面向制度性知识交付的 AI Skill 专业化形式，是包含以下七要素的最小自包含知识包：
  1. Intent Declaration（意图声明）
  2. Procedural Knowledge（过程知识）
  3. Tool Bindings（工具绑定）
  4. Organizational Metadata（组织元数据）
  5. Governance Constraints（治理约束）
  6. Continuation Paths（Continuation 路径）
  7. Validators（确定性验证器）

**与 Codified Context 的关系**：
- 论文明确将 Vasilopoulos 的工作作为互补引用：Codified Context 提供了单项目、单开发者的经验基础；Knowledge Activation 提供形式化框架、企业治理架构和组织级部署模型。
- 两者都强调：上下文不是临时提示，而是可维护的、机器可消费的基础设施。

**企业部署证据**：
- 在 Yahoo 部署了 87 个模块化、代理可消费的技能。
- 对 67 名工程师的匿名调查显示：四种感知体验驱动因素均有统计显著且大至非常大的效应量；每位工程师每周平均节省 2.6 小时；NPS +35；不同业务单元/ tenure 之间无显著差异。

**实践启示**：
- 企业应将制度性知识架构化，而非仅投资模型能力。
- AKU 不是文档片段，而是“行动就绪的规范”：告诉代理做什么、用什么工具、遵守什么约束、下一步去哪里。
- 维护是长期挑战：知识 commons 模型有助于可持续维护。

#### 10. Formal Architecture Descriptors as Navigation Primitives for AI Coding Agents（Jin, 2026）

**核心命题**：AI 编程代理把大量 tool call 花在无目的的代码库探索上。提供**形式化架构描述**能否减少这种导航开销？

**方法论亮点**：
- 三项互补研究：对照实验、artifact-vs-process 实验、观察性现场研究。
- 提出 **intent.lisp**：S-expression 架构描述格式，以及开源 **Forge toolkit**。

**关键发现**：
1. **架构上下文减少导航步骤 33–44%**（24 个代码定位任务 × 4 条件，Claude Sonnet 4.6，p=0.009，d=0.92）。
2. **格式对 LLM 理解无显著差异**：S-expression、JSON、YAML、Markdown 在 20 个问题 × 4 格式实验中均达到 95% 准确率。
3. **自动生成描述本身就有价值**：在 15 个任务 × 3 条件实验中，自动生成的描述达到 100% 准确率，盲搜仅 80%（p=0.002，d=1.04），证明描述文件价值不依赖“开发者自我澄清”效应。
4. **现场研究**：7,012 个 Claude Code 会话中，形式化声明与 **52% 代理行为方差降低** 相关。
5. **Writer-side 错误恢复模式不同**：
   - JSON：最低静默损坏（21%），但结构错误时原子性崩溃
   - YAML：50% 静默损坏
   - S-expression：检测所有结构完整性错误，但错误定位不精确

**为什么选 S-expression**：
- 不是因为它更容易被 LLM 理解（格式无差异），而是因为：
  - 语法强制层级结构
  - 优雅降级（非原子性失败）
  - 压缩密度高（生产项目 5:1–64:1，加权平均 34:1）
  - 支持下游工具链

**实践启示**：
- 上下文工程的辩论不应聚焦“LLM 偏好什么格式”，而应聚焦“什么格式在 LLM 同时读写时避免灾难性失败”。
- 形式化架构描述可以作为导航原语，显著降低代理探索成本。
- 自动生成描述可能优于人工精修的长描述（因为 token 预算竞争）。

#### 11. Harness-Native Software Engineering: The Control Plane of Coding Agents（Mishra, 2026）

**核心命题**：编码代理系统越来越由基础模型之外的“harness”决定：仓库指导、工具权限、执行沙箱、持久记忆、验证循环、恢复机制、委托逻辑。对于长程软件任务，这些设计选择往往决定代理能看到什么、能做什么、能否从失败中恢复。

**核心贡献**：
- 提出 **Harness-Native Software Engineering** 作为分析框架。
- 将运行时层形式化为 **Agent Harness Control Plane**，包含八大功能：
  1. Context ingress（上下文输入）
  2. Action mediation（动作调解）
  3. Execution substrate（执行基质）
  4. State persistence（状态持久化）
  5. Verification and review（验证与审查）
  6. Recovery and debugging（恢复与调试）
  7. Delegation and coordination（委托与协调）
  8. Governance and audit（治理与审计）

**五大命题**：
1. **Harness sensitivity**：许多仓库级/长程编码任务中，harness 条件变化带来的效用差异可与换模型相当。
2. **Recovery dominance**：长程任务中，可恢复性成为验证成功的主要决定因素。
3. **Artifact externalization**：团队 increasingly 将代理行为编码到版本控制工件中（manifest、skills、custom agents、memory）。
4. **Boundary displacement**：有效安全边界已从模型表面转移到调节工具、协议、权限和执行的 harness 接口。
5. **Evaluation inversion**：低估或不真实固定 harness 条件的基准测试会系统性误估真实部署能力。

**与上下文工程的关系**：
- 论文明确指出：context engineering 只是 Harness-Native Software Engineering 的一个切片；后者还包含审批策略、检查点、执行基质、审查循环、协议攻击面、多代理协调等。
- 该论文大量引用并讨论了 AGENTS.md 效率研究、AGENTS.md 负面评估、Codified Context 等上下文工程文献，将其作为 harness 成为“软件工件”的证据。

**实践工具**：
- 提出 **Harness Condition Sheet**：用于报告基准测试或部署研究中的具体 harness 条件，至少包括：基础模型、接口模式、上下文工件、工具/协议绑定、权限策略、执行基质、持久化特性、验证栈、恢复机制、委托拓扑、预算约束。

**局限**：概念/方法论论文，未报告新对照实验；证据混合官方文档、工程报告和近期预印本。

### C. 代码特异性上下文技术（新增）

> 本组 5 篇论文聚焦 AI Coding 场景：它们回答的是“在真实代码库中，**上下文从哪里来、如何被检索、如何被消费、如何被验证**”。它们与 A 组（为什么不能堆上下文）和 B 组（如何组织上下文基础设施）形成互补，共同构成上下文工程在软件工程领域的完整图景。

#### 12. SWE-bench: Can Language Models Resolve Real-World GitHub Issues?（Jimenez et al., 2024）

**核心命题**：现有代码生成基准（如 HumanEval）多为自包含的小问题，无法反映真实软件工程所需的跨文件推理、长上下文处理与执行交互能力。需要什么样的基准才能评估 LM 解决真实 GitHub issue 的能力？

**关键设计**：
- 从 12 个流行 Python 仓库的约 90,000 个 PR 中，通过三层过滤得到 **2,294 个任务实例**：
  1. 属性过滤：PR 必须解决一个 GitHub issue，且修改了测试文件；
  2. 执行过滤：PR 能成功安装，且应用后有测试从 fail 变为 pass；
  3. 最终保留可执行、可验证的实例。
- 任务输入：issue 文本描述 + 完整代码库快照；
- 任务输出：一个 patch，修改代码库以解决问题；
- 评估方式：执行仓库原有测试与 PR 引入的新测试。

**关键发现**：
- SWE-bench 任务通常需要跨多个函数、类甚至文件协调修改。
- 使用 BM25 检索器时，Claude 2 仅能解决 **1.96%** 的 issue；即使使用 oracle retriever，Claude 2 也仅提升到 4.8%。
- 模型在上下文长度增加时性能下降，且容易被额外上下文分散注意力（与 Liu et al. 的 U 型曲线结论一致）。
- 发布 SWE-bench-train（19,000 实例）与微调模型 SWE-Llama 7B/13B。

**对上下文工程的意义**：
- SWE-bench 将“上下文充足性”从抽象概念变为可测量指标：不是给了模型整个仓库就行，关键是它能否从巨大仓库中定位并消费正确上下文。
- 它证明了**检索质量**和**执行验证**是上下文工程在 AI Coding 中不可拆分的两翼。
- 后续大量工作（SWE-agent、Agentless、RepoGraph 等）都以它为共同评估基准。

#### 13. RepoGraph: Enhancing AI Software Engineering with Repository-level Code Graph（Ouyang et al., 2025）

**核心命题**：文件级 RAG 只能找到语义相似但不一定真正相关的代码；Agent 框架又容易在局部文件中陷入最优。能否用**细粒度的代码结构图**为 LM 提供仓库级导航？

**核心设计**：
- **行级代码图**：节点是代码行，边包括“定义-引用”“调用”“包含”等关系。
- **Ego-graph 子图检索**：以关键词（函数、类、issue 中的实体）为中心，抽取其局部依赖子图。
- **即插即用**：可同时接入 procedural 框架（定位-编辑两步）和 agent 框架（新增 `search_repograph` 动作）。

**关键发现**：
- 在 SWE-bench 上接入 4 种不同方法，**平均相对提升 32.8%**。
- 在 agent 框架中，新增 `search_repograph` 动作后：
  - 完成任务的最大回合数从 38 降至 35；
  - 平均回合从 21.47 降至 19.12；
  - 该动作多在前 15 轮被调用，比 `search_dir`/`search_file`/`find_file` 更精确。
- 在 CrossCodeEval 上也验证了可迁移性。

**错误分析**：
- 失败案例主要分为三类：
  1. **Incorrect localization**：未准确定位到代码片段；
  2. **Contextual misalignment**：生成的 patch 与代码库整体上下文不一致；
  3. **Regressive fix**：修复原问题时引入新问题。

**实践启示**：
- 代码库上下文不应只是文本片段，而应是有结构的图；**结构关系本身是一种上下文**。
-  ego-graph 检索能在保持上下文可控的同时，提供跨文件依赖信息。
- 这与 B 组 Jin 的“形式化架构描述”和 Bakal 的“AKU”形成互补：RepoGraph 提供的是**自动化生成的代码结构上下文**，而非人工撰写的规范。

#### 14. SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering（Yang et al., 2024）

**核心命题**：LM agent 作为新的“终端用户”，需要专门为其设计的计算机接口，而非直接复用为人类设计的 Linux shell 或 IDE。如何设计 Agent-Computer Interface（ACI）以提升 AI 软件工程能力？

**核心设计**：
- 提出 **Agent-Computer Interface（ACI）**：介于 LM agent 与计算机之间的抽象层，定义命令集合与环境反馈格式。
- SWE-agent 的 ACI 包含四大组件：
  1. **搜索/导航**：`find_file`、`search_file`、`search_dir`（限制返回数量，避免信息过载）；
  2. **文件查看器**：按窗口展示文件内容，支持跳转；
  3. **文件编辑器**：支持行级编辑，并给出语法检查等 guardrails；
  4. **上下文管理**：控制历史回合的格式与长度。

**ACI 设计原则**：
1. **动作简单易懂**：减少选项和文档长度；
2. **动作紧凑高效**：把常见操作（如导航+编辑）合并为少量动作；
3. **反馈信息丰富但简洁**：告诉 agent 动作结果，但不堆砌无关细节；
4. **Guardrails 防止错误传播**：如编辑时自动语法检查、给出错误提示。

**关键发现**：
- 使用 GPT-4 Turbo，SWE-agent 在 SWE-bench 上达到 **12.47%** 的 pass@1，远超此前非交互式 RAG 系统的 3.8%。
- 在 SWE-bench Lite 子集上的消融：相比仅使用默认 Linux shell 的 baseline，ACI 带来 **10.7 个百分点**的提升。
- ACI 可迁移到不同 LM：Claude 3 Opus 版本达到 10.5%。
- 在 HumanEvalFix 上达到 87.7%。

**对上下文工程的意义**：
- ACI 本质上是一种**受控的上下文输入/输出协议**：它决定 agent 能看到什么反馈、能以什么动作改变环境。
- 它把“上下文工程”从静态文件组织扩展到**动态交互协议设计**。
- 与 Mishra 的 Harness Control Plane 相呼应：ACI 是 harness 中“Action mediation”与“Context ingress”的具体实现。

#### 15. Agentless: Demystifying LLM-based Software Engineering Agents（Xia et al., 2024）

**核心命题**：复杂的自主 agent（多轮工具调用、自主规划）真的是解决真实软件工程问题的最佳路径吗？一个**简单、受控、无 agent** 的三段式流程能否取得更好效果？

**核心设计**：
Agentless 采用 **定位（Localization）→ 修复（Repair）→ 验证（Patch Validation）** 三阶段：
1. **定位**：
   - 层次化定位：先定位到文件，再定位到类/函数，最后定位到具体编辑位置；
   - 结合 LLM-based 定位与经典信息检索（embedding-based）定位。
2. **修复**：
   - 在定位的编辑位置生成多个候选 patch（diff 格式）；
   - 同时生成 reproduction test，用于验证问题是否被复现。
3. **验证**：
   - 用 regression tests + 生成的 reproduction tests 对候选 patch 重排序并选择最终提交。

**关键发现**：
- 在 SWE-bench Lite 上，Agentless 达到 **32.00%**（96/300 正确修复），超过当时所有开源 agent-based 方法。
- 成本极低：平均每个任务仅 **$0.70**。
- 已被 OpenAI 用于展示 GPT-4o 和 o1 模型的真实编码性能。
- 人工分析发现 SWE-bench Lite 中存在数据质量问题：4.3% 的 issue 描述中包含完整 ground-truth patch，10.0% 缺少关键信息，5.0% 有误导性描述。据此构造更严格的 **SWE-bench Lite-S**。

**对上下文工程的意义**：
- Agentless 提示我们：**上下文工程不等于 agent 工程**。受控的上下文消费流程（定位→修复→验证）可能比让 LLM 自主决定每一步更可靠。
- **执行反馈（测试、reproduction）是上下文的一部分**：Agentless 把测试结果作为筛选 patch 的上下文信号。
- 它与 SWE-agent 形成鲜明对照：
  - SWE-agent 强调“交互接口设计”，Agentless 强调“流程控制”；
  - 两者共同说明：harness/流程设计对最终效果的影响可与换模型相当。

#### 16. CodeRAG-Bench: Can Retrieval Augment Code Generation?（Wang et al., 2024）

**核心命题**：RAG 在文本任务中已被证明有效，但在代码生成中是否同样有效？什么情况下检索能真正帮助代码生成？

**关键设计**：
- 构建 **CodeRAG-Bench**：覆盖 6 个编程任务、4 个类别（基础编程、开放域编程、仓库级、代码检索）。
- 聚合 **5 类检索来源**：竞赛题解、在线教程、Python 库文档、StackOverflow 帖子、GitHub 文件。
- 总共 **9K 编程任务 + 25M 检索文档**，并为每个任务人工标注 canonical documents。

**关键发现**：
- 高质量上下文确实能显著提升代码生成：
  - GPT-4o 在 SWE-bench 上获得 **27.4%** 的绝对提升；
  - 在更难的 ODEX 子集上获得 **6.9%** 提升。
- 但当前检索器仍难以获取真正有用的上下文，尤其在开放域和仓库级任务中。
- 生成器在利用有限上下文和整合额外上下文方面存在局限；更多文档不一定带来更好性能（与 A 组长上下文失效研究一致）。

**对上下文工程的意义**：
- CodeRAG-Bench 把“代码上下文检索”作为一个独立研究问题提出来，并给出可复现的评估框架。
- 它验证了上下文工程的核心矛盾：**信息存在 ≠ 信息可用 ≠ 信息有用**。即使能检索到相关文档，模型也不一定能有效利用。
- 与 RepoGraph 形成对照：CodeRAG-Bench 关注“检索什么外部/内部文档”，RepoGraph 关注“抽取什么代码结构关系”。两者共同覆盖代码上下文的两大来源。

#### 17. Codebase-Memory: Tree-Sitter-Based Knowledge Graphs for LLM Code Exploration via MCP（Vogel et al., 2026）

**核心命题**：当前 LLM coding agent（如 Claude Code、Cursor、Aider）主要通过反复读取文件和 grep 搜索来探索代码库，每次查询消耗数千 token，却仍未建立结构性理解。能否把代码库结构预先解析为持久化知识图谱，通过 MCP 以极低成本向 agent 暴露结构化查询能力？

**关键设计**：
- **Tree-Sitter 解析**：支持 **66 种编程语言**，从 AST 中提取定义（函数、方法、类、接口、枚举、类型）、调用点、导入、引用、trait 实现等。
- **混合类型解析**：对 Go、C、C++ 结合 LSP 风格的类型解析，提升方法接收器、指针间接引用、包限定标识符的调用图精度。
- **多阶段并行构建管道**：使用 pthreads 工作池并行提取，先写入内存图缓冲区，再批量 flush 到 SQLite；支持增量同步（文件监听 + XXH3 内容哈希重索引）。
- **MCP 工具接口**：暴露 **14 类结构化查询工具**，包括：
  - 索引类：名称/符号搜索、定义查找；
  - 查询类：调用路径追踪、影响分析、hub 检测、死代码检测；
  - 分析类：社区发现（Louvain）、复杂度统计；
  - 代码类：Cypher 查询、代码片段获取。
- **跨服务调用检测**：针对 Python、Go、Java/Spring、Kotlin/Ktor、Express.js、Laravel 提供 6 种框架特定的 HTTP 调用/路由提取器，支持微服务架构。
- **零依赖部署**：单静态链接 C 二进制，内置 66 个 Tree-Sitter 语法，状态仅存于一个 SQLite 文件。

**关键发现**：
- 在 **31 个真实仓库**上评估，与基于文件探索的 agent 对比：
  - 回答质量：**83%** vs **92%**（文件探索基线）；
  - Token 消耗：仅为文件探索的 **1/10**；
  - Tool call 次数：减少 **2.1 倍**。
- 对于图原生查询（hub 检测、caller ranking 等），在 **19/31 种语言**上匹配或超过文件探索 agent。
- 索引性能：Linux kernel（28M LOC，75K 文件）3 分钟内完成，生成 4.81M 节点、7.72M 边；查询延迟亚毫秒。
- 典型结构问题 token 节省示例：
  - 函数模式查找：~200 vs ~45,000（225×）；
  - 深度 3 调用链追踪：~800 vs ~120,000（150×）；
  - 死代码检测：~500 vs ~85,000（170×）；
  - 五个查询总计：~3,400 vs ~412,000（~121×）。

**对上下文工程的意义**：
- **把“代码结构”从隐式文本关系提升为显式、可查询的第一类上下文**。这与 RepoGraph 的代码图思路一脉相承，但 Codebase-Memory 更进一步：通过 MCP 将图谱作为持久化、增量维护、零依赖的基础设施暴露给任何兼容 agent。
- **验证了“结构化上下文”可以显著降低 token 成本**，即使回答质量略有下降（83% vs 92%），在多数结构性问题场景下仍是更优的上下文工程策略。
- **MCP 成为上下文交付协议**：它不是把知识图谱作为一次性输入塞进 prompt，而是让 agent 按需调用结构化工具获取上下文——这与 Bakal 的 AKU、Vasilopoulos 的 MCP 知识库在理念上高度一致。
- **与现有工作正交**：可与 SWE-agent、Agentless、OpenHands 等任何 agent 架构结合，只需替换/增强其检索层。

## 三、横向比较与核心主题提炼

### 3.1 从“长上下文极限”到“上下文工程”：必要性论证

A 组 6 篇文献从多个角度共同证明：

> **LLM 厂商宣称的上下文窗口是一种“架构上限”，而非“有效工作记忆”。模型实际可稳定利用的上下文长度通常只有宣称值的 30%–60%，且存在任务依赖性。**

| 文献 | 核心度量 | 关键数值/发现 |
|:---|:---|:---|
| Context Is What You Need | MECW | 多数模型在 1,000 tokens 内开始严重退化 |
| Lost in the Middle | 位置偏置 | 中间位置性能甚至低于“闭卷”基线 |
| Intelligence Degradation | 临界阈值 | Qwen2.5-7B 在 43.2%（~55K）处断崖式下跌 |
| Limitations of Normalization | 注意力稀释 | 仅 ~6% 的 token 被区分后即趋均匀分布 |
| Fundamental Limits | 有效上下文增长 | 亚线性增长；Llama3.1 有效利用仅约 64K/128K |
| Context Rot | 多任务退化 | 18 个模型全部表现出某种形式的 Context Rot |

这些发现共同说明：**不能依赖“堆更多上下文”解决复杂问题**。这正是上下文工程作为独立学科兴起的前提。

### 3.2 失效机制的多层次解释

这些文献构建了一个从微观到宏观的多层次失效解释框架：

1. **数学层（微观）**：Softmax 归一化的长度无关性导致注意力权重随 L 增长而均匀化（Mudarisov et al.）。
2. **机制层（中观）**：位置编码（正弦/RoPE）的长程衰减、注意力熵增、远端位置的梯度欠训练（Mohsin et al.; Wang et al.）。
3. **行为层（宏观）**：U 型曲线、临界阈值、Context Rot、级联失效（Liu et al.; Paulsen; Hong et al.）。

### 3.3 上下文工程的演化：从“提示技巧”到“基础设施”

B 组 5 篇文献展示了上下文工程如何从个人/项目层面的提示技巧，演化为可规模化的基础设施：

| 阶段 | 代表文献 | 核心特征 |
|---|---|---|
| **单文件 manifest** | Lulla et al. | 一个 `AGENTS.md` 即可带来 28% 时间节省和 16% token 节省 |
| **单项目多层架构** | Vasilopoulos | 热内存宪法 + 专家代理 + 冷知识库，通过 MCP 检索 |
| **企业级知识原语** | Bakal | 基于 Agent Skills 标准的 AKU，强调治理、验证、组织元数据 |
| **形式化架构描述** | Jin | intent.lisp 作为导航原语，自动/人工生成描述减少 33–44% 探索步骤 |
| **Harness 控制平面** | Mishra | 将上下文工程置于更广阔的 harness 视角，提出八大控制功能和 Harness Condition Sheet |

这个演化路径可以概括为：

> **上下文工程正在从“怎么写 prompt”扩展为“如何设计、维护、治理机器可读的项目/组织知识基础设施”。**

### 3.4 代码特异性上下文技术：AI Coding 场景不可或缺的第三支柱（新增）

C 组 5 篇论文将上下文工程的研究对象从“通用文本上下文”推进到“真实代码库上下文”：

| 论文 | 解决的问题 | 核心方法 | 关键结论 |
|---|---|---|---|
| SWE-bench | 如何评估真实代码库级能力 | 真实 GitHub issue + 执行验证 | 当前模型解决率极低（Claude 2 仅 1.96%），上下文利用是瓶颈 |
| RepoGraph | 如何表示和利用代码库结构 | 行级代码图 + ego-graph 检索 | 结构化代码图平均提升 32.8%，减少 agent 探索回合 |
| SWE-agent | 如何为 LM 设计交互接口 | Agent-Computer Interface（ACI） | 专用接口比 Linux shell 提升 10.7 个百分点 |
| Agentless | 复杂 agent 是否必要 | 定位-修复-验证三段式 | 简单受控流程达到 32% 且成本仅 $0.70/任务 |
| CodeRAG-Bench | 代码 RAG 是否有效 | 9K 任务 + 25M 文档的基准 | 高质量上下文有效，但检索器和生成器利用能力都有限 |
| Codebase-Memory | 如何把代码结构作为持久化、低成本的上下文基础设施 | Tree-Sitter + SQLite + MCP，66 种语言 | Token 成本降至 1/10，tool call 减少 2.1 倍，83% 回答质量 |

它们共同揭示：在 AI Coding 中，上下文工程必须同时处理三类上下文：
1. **外部知识上下文**（文档、StackOverflow、库说明）——CodeRAG-Bench；
2. **内部结构上下文**（代码依赖、调用关系、模块边界）——RepoGraph；
3. **执行反馈上下文**（测试结果、错误信息、reproduction）——SWE-agent、Agentless。

### 3.5 关键共识与张力

#### 共识
1. **上下文质量 > 上下文长度**：关键不是窗口多大，而是里面放了什么、怎么组织。
2. **结构化优于非结构化**：表格、显式约束、工具绑定、验证器比自然语言散文更可靠。
3. **维护是关键风险**：stale specs 会误导代理；上下文工件需要版本控制和持续更新。
4. **代理行为可被外部化工件塑造**：`AGENTS.md`、skills、context docs、architecture descriptors 都能显著改变代理行为。
5. **执行反馈是上下文的一部分**：测试、错误信息、reproduction 结果应被纳入上下文循环（SWE-agent、Agentless）。
6. **代码结构是一种上下文**：代码图、依赖关系、调用链与文本描述同等重要（RepoGraph、Codebase-Memory）。
7. **上下文基础设施应持久化、增量维护、零依赖**：Codebase-Memory 通过 SQLite + MCP 将代码知识图谱变成可随时查询、低成本的上下文服务。

#### 张力
1. **上下文文件可能帮助效率但损害成功率**：Lulla et al. 发现 `AGENTS.md` 提升效率；Gloaguen et al.（被 Bakal 和 Jin 引用）发现朴素上下文文件可能降低成功率并增加成本。这说明**质量与结构至关重要**。
2. **格式之争尚无定论**：Jin 的实验显示 LLM 理解层面格式无显著差异；但不同格式在 writer-side 错误恢复、压缩密度、工具链支持上差异显著。
3. **单项目经验 vs. 企业规模化**：Vasilopoulos 的 N=1 经验报告提供了详细案例，但缺乏严格因果证据；Bakal 提供了企业调查证据，但细节披露有限。
4. **Agent vs. Agentless**：SWE-agent 证明专用交互接口的价值；Agentless 证明受控流程可以更简单、更便宜、更有效。两种范式并非互斥，而是适用于不同 harness 设计。
5. **检索增强 vs. 结构增强**：CodeRAG-Bench 强调从外部/内部文档中检索；RepoGraph 强调从代码结构中抽取。真实系统需要两者的融合。

### 3.6 对上下文工程的定义补充

综合全部 16 篇文献，可以对“上下文工程”给出更精确的定义：

> **上下文工程是在 LLM 的“有效工作记忆”边界内，通过信息的选择、排序、结构化、压缩、路由、检索和治理，最大化模型对关键信息的注意力利用效率，并将项目/组织知识以及代码库结构转化为可维护、可验证、可机器消费的基础设施，从而补偿模型长上下文处理能力的固有缺陷。**

其核心原则包括：
- **长度控制**：单次输入应控制在模型 MECW 的 40%–60% 以内。
- **位置优化**：将最关键信息置于输入开头和结尾。
- **干扰过滤**：主动移除语义相近的干扰信息。
- **结构简化/形式化**：使用机器可解析的结构（S-expression、JSON、YAML、Markdown 表格、代码图）表达架构和约束。
- **任务解耦**：将检索与推理分离，将探索与编辑分离，将定位-修复-验证分离。
- **知识分层**：区分热内存（总是加载）、专家代理（按任务调用）、冷知识库（按需检索）。
- **治理嵌入**：在知识工件中嵌入权限、验证器、审计轨迹，而不仅仅是操作说明。
- **持续维护**：将上下文工件视为“load-bearing infrastructure”，与代码同步演进。
- **执行反馈闭环**：将测试结果、错误信息、reproduction 结果作为动态上下文注入。
- **代码结构显式化**：利用 AST、调用图、依赖图等代码结构作为上下文来源。

## 四、实践建议矩阵

| 场景 | 推荐做法 | 参考文献 |
|---|---|---|
| 小型项目（<10K 行） | 先写一个 `AGENTS.md` 或 `CLAUDE.md`，包含项目目标、技术栈、核心约定 | Lulla et al. |
| 中型项目（10K–100K 行） | 采用 Vasilopoulos 三层架构：constitution + specialist agents + MCP 知识库 | Vasilopoulos |
| 大型企业/多团队 | 采用 Bakal 的 AKU 框架，基于 Agent Skills 标准，强调治理、验证、组织元数据 | Bakal |
| 需要严格架构导航 | 引入 Jin 的 intent.lisp/Forge，用形式化架构描述减少代理探索 | Jin |
| 评估/基准测试 | 使用 Mishra 的 Harness Condition Sheet 明确报告 harness 条件；用 SWE-bench 评估代码库级能力 | Mishra, Jimenez et al. |
| 需要从代码库抽取结构上下文 | 构建 RepoGraph 风格的行级代码图，用 ego-graph 检索跨文件依赖 | Ouyang et al. |
| 需要为 LM 设计交互接口 | 参考 SWE-agent 的 ACI 设计原则：简单动作、紧凑反馈、guardrails | Yang et al. |
| 需要低成本高可控的修复流程 | 采用 Agentless 的定位-修复-验证三段式，结合 regression + reproduction tests | Xia et al. |
| 需要评估/改进代码 RAG | 使用 CodeRAG-Bench 框架，关注检索质量与生成器利用能力 | Wang et al. |
| 需要把代码结构作为低成本上下文服务 | 部署 Codebase-Memory 类系统：Tree-Sitter 解析 + SQLite + MCP，支持 66 种语言 | Vogel et al. |
| 任何规模 | 控制单次上下文长度，监控 attention decay，定期清理 stale specs，将执行反馈纳入上下文循环 | A 组全部，SWE-agent，Agentless |

## 五、结论

这 17 篇文献共同描绘了一幅从“问题诊断”到“基础设施构建”再到“代码特异性技术”的完整图景：

1. **LLM 的长上下文能力存在根本性、多维度的瓶颈**（A 组），这些瓶颈根植于注意力机制的数学性质、位置编码的物理限制和训练分布的统计偏置。
2. **上下文工程应运而生**：不是简单写 prompt，而是系统性地管理模型看到什么、何时看到、以什么格式看到。
3. **上下文正在基础设施化**：从单文件 `AGENTS.md`（Lulla et al.），到单项目三层上下文架构（Vasilopoulos），再到企业级原子知识单元（Bakal）、形式化架构描述（Jin）和 harness 控制平面（Mishra）。
4. **代码特异性上下文技术成为第三支柱**：SWE-bench 提供了真实评估场景，RepoGraph 提供了代码结构上下文，SWE-agent 与 Agentless 提供了两种不同的高效 harness 范式，CodeRAG-Bench 为代码 RAG 提供了系统评估框架，Codebase-Memory 则把代码知识图谱变成持久化、低成本的上下文基础设施。
5. **证据强度逐渐增强**：Lulla 和 Jin 提供了严格对照实验；Vasilopoulos 提供了深度经验报告；Bakal 提供了企业调查；Mishra 提供了概念框架；C 组论文则在真实代码库任务上提供了大量可量化的证据。

对于工程实践而言：
- **不要盲目信任厂商的上下文窗口宣称**——有效窗口通常只有宣称值的一半甚至更少。
- **不要依赖“堆更多上下文”来解决复杂问题**——超过临界阈值后，增加上下文不仅无益，反而有害。
- **将上下文工程视为与提示工程同等重要的独立学科**——它涉及信息检索、排序、压缩、结构化、路由、治理、执行反馈和系统级设计。
- **把上下文工件当作基础设施来维护**——它们是 load-bearing artifacts，需要版本控制、审查、测试和持续更新。
- **在 AI Coding 场景中，要特别关注代码结构上下文和执行反馈上下文**——仅有文本描述和文档是不够的。

未来的上下文工程研究应聚焦于：
- 为更多模型建立 MECW 与临界阈值的公开数据库；
- 开发动态上下文预算分配算法；
- 探索长度感知注意力机制和迭代式 planner-reader 架构；
- 建立上下文工件的质量评估标准（如 knowledge density ρ、stale-spec 检测）；
- 推广 Harness Condition Sheet 等评估协议，使基准测试能区分模型效应与 harness 效应；
- 融合代码图检索与文档 RAG，构建统一的代码上下文检索框架；
- 研究如何将测试/执行反馈自动、可靠地注入上下文循环。
- 探索 RepoGraph 的“子图注入”与 Codebase-Memory 的“MCP 按需查询”两种范式的融合。
